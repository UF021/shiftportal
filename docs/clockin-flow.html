Override Path — Manager Authorisation
🔑 Override Form
• Duty Manager full name (required)

• Reason: GPS unavailable / arrived early
  / other
POST /clock/{slug}/{code}/in|out
manager_override: true · no GPS · manual_time allowed
Same staff lookup + lateness calc
No GPS check — override bypasses geofence
💾 INSERT ClockEvent
entry_notes = "[OVERRIDE] Manager: {name} | Reason: {reason}"
Why overrides exist:
GPS may be blocked indoors, in basements, or on certain handsets. The override path lets a duty manager on-site authorise a clock event without geofencing — the manager's name and reason are permanently stored in entry_notes so HR can audit every bypass.
Override badge in HR view:
HRTimelogs parses [OVERRIDE] prefix from entry_notes and shows an amber ⚠ Override badge alongside the manager's name.
What overrides cannot bypass:
Staff ID / name lookup still runs — an override does not allow clocking in as someone else. Only GPS geofencing is bypassed.
Error States Reference
🔐 Auth errors
• Staff ID not recognised (401)
• Name not recognised (401)
• Account not activated (403)
📍 Location errors
• GPS denied by browser
• GPS timeout (>15 s)
• Too far from site (>70 m)
⚡ State errors
• Already clocked in (409)
• No active clock-in (400)
• Network error (catch)
What gets stored in the database
ClockEvent (clock_in row)
• event_type: clock_in
• timestamp: UTC datetime
• user_id, site_id, organisation_id
• scheduled_start: HH:MM string
• is_late: boolean
• minutes_late: integer
• gps_lat, gps_lng, gps_verified
• entry_notes: override info or null
ClockEvent (clock_out row)
• event_type: clock_out
• timestamp: UTC datetime
• user_id, site_id, organisation_id
• shift_minutes: total duration
• gps_lat, gps_lng, gps_verified
Paired to clock_in by: clock_out.timestamp > clock_in.timestamp
